# 研发路线

## 近期目标

项目近期目标是从文档基线进入可运行 ROS2 Humble 工作区，并逐步完成硬件、仿真、感知、导航、采样和端到端联调。

团队分工、接口责任和分支策略见 [../engineering/team_workflow.md](../engineering/team_workflow.md)。研发推进时按成员负责的可验收能力检查进度，而不是按代码量检查进度。

车机从 Ubuntu 20.04 + ROS Noetic 过渡到 Ubuntu 22.04 + ROS2 Humble 的闭环路线见 [车机 ROS2 迁移技术路线](vehicle_ros2_migration_route.md)。该路线优先采用可回滚、可验证、逐模块替换策略，不直接把厂商 `tzb2026/` 代码整体改成 ROS2。

## 当前状态

- 阶段 0 初始化: 基础文档和 ROS2 工作区骨架已完成，`colcon build --symlink-install` 已于 2026-06-25 在 Ubuntu 22.04 + ROS2 Humble 环境中确认通过。
- 当前硬件状态: SCOUT MINI、PiPER 和传感器实物已到；传感器线束已接入车机并上胶固定，本机不具备直接连接传感器进行实测的条件。
- 车机 Noetic 基线: 2026-07-01 记录，已按 `tzb2026/readme-2451.txt` 在车机执行厂商 ROS Noetic 流程测试，当前反馈为无问题；该结论仅覆盖厂商原始 Noetic 流程，不等同于 ROS2 Humble 硬件 bringup 已完成。
- 当前推进路径: 本机继续推进 ROS2 主线、mock/sim、构建和静态验证；涉及真实底盘、PiPER、雷达、相机、IMU 的验证需要将代码部署到车机或 ROS2 主控后执行。
- Mock/sim 状态: 已验证 mock 目标、mock 导航状态、基础传感器 topic、任务 action 和 rosbag 记录，详见 [../validation/mock_bringup_validation.md](../validation/mock_bringup_validation.md)。
- 阶段 1 硬件 Bringup 状态: ROS2 Humble 实车 bringup 待部署到车机验证，不视为已完成。
- 仍需保持待验证状态: 重量、尺寸、传感器外参、支架方案、第三方 ROS2 驱动 commit 和 ROS2 硬件实测指标。

## 阶段 0: 初始化

目标:

- 建立 README、协作规则和核心设计文档。
- 建立 ROS2 Humble 单仓库工作区骨架、接口包、bringup 包、description 包和任务状态机包。
- 建立 7 人任务划分、跨模块接口约定和代码分支策略。
- 明确赛题硬约束、平台基线和一级风险。
- 记录待归档资料来源。

验收:

- 文档可以支撑后续代码、依赖和硬件任务拆分。
- 每个成员有明确主责、交付物、接口消费者和验收标准。
- `colcon build --symlink-install` 已于 2026-06-25 在 Ubuntu 22.04 + ROS2 Humble 环境中确认通过。
- 未验证事实均标注待验证、待归档或待称重。

## 阶段 1: 硬件 Bringup

目标:

- 在 Ubuntu 22.04 + ROS2 Humble + Python 3.10 环境中启动底盘、机械臂和传感器。
- 锁定 `scout_ros2` 与 `piper_ros` humble 的来源、分支和 commit。
- 建立基础 TF、URDF 和启动文件。

验收:

- 底盘可发布里程计并响应低速 `cmd_vel`。
- PiPER 可发布关节状态并执行安全预设动作。
- RGB-D、RoboSense RSHELIOS_16P 雷达、IMU 数据稳定发布。
- 完成基础 rosbag 和 bringup 记录。

责任人:

- A 负责 SCOUT MINI 底盘。
- C 负责 LiDAR、IMU、TF 和基础建图。
- F 负责 PiPER 上电、关节状态和安全预设动作。

## 阶段 2: 仿真

目标:

- 建立简化机器人模型、场地模型和科学目标模型。
- 在仿真中验证状态机、Nav2 参数和采样动作流程。
- 尽量复用真实 topic 和 launch 结构。

验收:

- 可在仿真中完成离开基地、搜索点巡航、接近目标、返回基地的流程。
- 任务状态机具备超时、失败和恢复路径。

## 阶段 3: 目标识别

目标:

- 构建科学目标检测与 3D 位姿估计流程。
- 评估补光灯对识别稳定性的影响。
- 输出可抓取性判断。

验收:

- 在典型距离和角度下稳定输出目标类别、置信度和位姿。
- 能处理漏检、误检、遮挡和深度异常。
- 识别结果可被导航和操控模块消费。

责任人:

- D 负责目标检测、补光和置信度。
- E 负责 3D 位姿、坐标转换和 `/target_detections`。

## 阶段 4: 导航与越障

设计入口: [探索导航避障设计方案](../design/navigation_planning.md)。

目标:

- 完成 slam_toolbox、robot_localization、Nav2 参数调试。
- 建立基地、搜索区、目标接近点和放置区的导航策略。
- 针对比赛场地障碍和光照条件调试局部避障。

验收:

- 机器人能稳定离开基地并到达搜索点。
- 能在 600 秒约束下规划返回基地路径。
- 导航失败时能执行有限恢复动作。

责任人:

- B 负责 Nav2、局部避障和返回基地策略。
- A/C 保障底盘、里程计、TF、雷达和定位输入稳定。

## 阶段 5: 采样

目标:

- 建立 PiPER 预抓取、抓取、抬升、收回和放置动作序列。
- 验证末端执行器对科学目标的夹持稳定性。
- 将目标位姿转换为机械臂抓取位姿。

验收:

- 机械臂能在静态目标上重复完成抓取和放置。
- 底盘与机械臂动作互锁，避免移动中误操作。
- 抓取失败后有重试和放弃策略。

责任人:

- F 负责 PiPER 抓取、携带和放置动作。
- E 负责把目标 3D 位姿转换为机械臂可用抓取候选。

## 阶段 6: 系统联调

目标:

- 打通离开基地、搜索、识别、取样、返回、放置全流程。
- 录制端到端 rosbag、视频和状态机日志。
- 固化比赛参数和启动流程。

验收:

- 至少完成多次端到端闭环演练。
- 每次演练记录耗时、失败点、恢复动作和改进项。
- 完成最终重量、尺寸、电源和安全检查。

## 阶段 7: 比赛材料

目标:

- 整理提交物、部署说明、现场检查单和备份方案。
- 归档官方资料、第三方依赖和关键参数版本。
- 准备两次出发机会的快速复位流程。

验收:

- 提交包内容完整。
- 现场人员可按检查单完成开机、校验、启动、复位和日志保存。
- 所有“待归档”“待称重”“待验证”项在赛前关闭或明确风险接受。
