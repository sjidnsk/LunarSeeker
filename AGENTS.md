# AGENTS.md

本文件定义本仓库内 Codex / agent 协作的固定规则。除非用户在当前任务中明确覆盖，否则所有 agent 必须遵守。

## 沟通语言

- 默认使用中文回答。
- 用户明确要求其他语言时，按用户指定语言回答。
- 文档和代码注释默认使用中文，除非项目上下文要求英文接口名、协议名或上游原文。

## 项目技术基线

- 系统固定为 Ubuntu 22.04。
- ROS 固定为 ROS2 Humble。
- Python 固定为 Python 3.10。
- 机器人平台基线为 SCOUT MINI + PiPER。
- 主要 ROS 组件基线包括 Nav2、slam_toolbox、robot_localization、ros2_control。
- 不要在未确认兼容性的情况下引入其他 ROS 发行版、Python 大版本或系统版本假设。

## 系统工程约束

- agent 必须优先维护系统边界、模块职责、公共接口和依赖声明的一致性，不得为了完成局部任务随意扩张主线代码。
- 修改前必须先搜索现有模块、入口、schema、消息、配置字段和依赖声明；已有合适实现时优先复用或小步扩展。
- 默认禁止一次性生成上万行大篇幅代码、配置或文档。
- 如果预计新增或修改超过约 500 行生产代码，必须先说明拆分方案、模块边界、风险和验证方式；未经用户确认不得直接堆叠实现。
- 不允许在未解释必要性的情况下引入新的框架、运行时、构建系统或大型依赖。
- 不允许为解决局部问题复制整套相似实现；发现重复实现时，应优先收敛到单一权威入口或明确标注临时状态。

### 主线模块与临时代码

- 新模块进入主线前必须明确职责、输入输出、依赖关系、公共接口和替代对象。
- 临时验证代码必须放在测试、mock、examples、tools 或明确的实验目录中，不得伪装成生产主线。
- 经过多轮修补后形成事实主线的模块，必须回收成清晰接口、文档和测试；不得继续堆叠补丁。
- 不允许同时保留多个功能重叠的入口、脚本、launch 文件或配置模板，除非文档说明各自适用场景。

### Schema、消息和字段定义

- ROS msg/action、配置 schema、特征字段、状态枚举和坐标系约定必须有唯一权威定义位置。
- 不允许在多个模块重复定义同名、近义或语义重叠字段。
- 新增字段前必须先搜索现有字段、消息、配置和文档。
- 字段定义处必须说明命名含义、单位、坐标系、取值范围和验证状态；未实测参数必须标注“待测量”或“待验证”。
- 如需兼容旧字段，必须标注 deprecated，并给出迁移路径和计划移除条件。

### 依赖管理

- 新增依赖必须同步更新对应的 `package.xml`、`setup.py`、`CMakeLists.txt`、`dependencies.repos`、安装说明或依赖文档。
- 不允许只在代码中 `import`、`find_package` 或调用外部命令而不声明依赖。
- 不允许引入与 Ubuntu 22.04、ROS2 Humble、Python 3.10 不兼容的依赖。
- 第三方源码不得直接进入主线目录，必须放入约定位置并记录来源、版本、许可证和验证状态。
- 同类功能已有依赖时，优先复用；新增替代依赖必须说明理由、收益和迁移成本。

### 入口、脚本和启动文件

- 每个常用工作流只能有一个推荐入口。
- 新增脚本、launch、命令封装时，必须说明它和已有入口的关系。
- 不允许出现多个名字不同但行为近似的启动脚本、调试脚本或配置入口。
- 废弃入口必须标注 deprecated，或在确认安全后删除。
- README、docs 和测试中的入口命令必须与实际文件保持一致。

### 测试约束

- 测试应优先验证公开接口、消息、行为、文件布局和端到端契约，不应锁死私有函数、内部字段、实现顺序或临时日志文本。
- 不允许为了通过测试而暴露内部状态、复制实现逻辑或把私有实现固化成公共契约。
- 修改 schema、入口、依赖、launch、配置或跨模块接口时，必须补充对应测试，或明确说明未测试原因。
- mock 测试不得伪造过度乐观的硬件结论；未实测内容必须标注“待测量”“待验证”。
- 测试失败时应优先修正行为、契约或测试目标，不得随意放宽断言掩盖真实回归。

### 大改动刹车规则

出现以下任一情况时，agent 必须先停止并说明方案，不得直接继续堆代码:

- 需要新增多个主线模块。
- 需要新增或修改超过约 500 行生产代码。
- 需要引入新的第三方依赖。
- 发现已有两个以上重复实现。
- 需要改变消息、action、配置 schema、公共入口或跨模块接口。
- 测试开始依赖私有函数、内部字段或实现顺序。

## 文件管理与危险操作

- 默认禁止批量删除文件或目录。
- 禁止使用以下命令或等价操作:
  - `del /s`
  - `rd /s`
  - `rmdir /s`
  - `Remove-Item -Recurse`
  - `rm -rf`
- 需要删除文件时，只能一次删除一个明确路径的文件，例如:

```powershell
Remove-Item "D:\path\to\file.txt"
```

- 如果确实需要批量删除文件或目录，必须停止并请求用户手动确认或手动删除。
- 递归移动、批量移动、大范围覆盖也按危险操作处理；执行前必须确认目标路径明确且范围受控。
- 允许读取文件、搜索文本、列目录、查看状态、运行测试和 dry-run 命令。

## 下载与临时资源位置

- 默认将下载资源、安装包、数据集、模型、缓存、导出文件等保存到 D 盘。
- 推荐路径:

```text
D:\CodexDownloads
```

- 不要默认把大文件、临时下载、数据集、模型、构建产物、Node cache、训练输出放在 C 盘。
- 小型测试产物、pytest cache、项目 ignored output 目录可以按工具默认行为生成，但不应提交进 Git。
- 如果 D 盘目录不存在，应优先创建明确的 D 盘目录，或在回复中说明准备使用的路径；不要静默改写到 C 盘。
- 如果由于工具或系统限制只能写入 C 盘，必须说明完整路径、用途、是否可以删除、建议删除时机。

## Git 与用户改动保护

- 默认认为工作区中已有改动可能来自用户或其他任务。
- 禁止执行:
  - `git reset --hard`
  - `git checkout -- <path>`
  - 大范围 revert
  - 清理未跟踪文件的批量命令
- 除非用户明确要求，不要回退、覆盖或删除已有改动。
- 提交时只纳入当前任务相关文件。
- 分支命名、用途和合入条件以 `docs/engineering/team_workflow.md` 的“开发分支”章节为准；agent 不应擅自创建、切换或重命名分支，除非用户明确要求。
- 发现无关 dirty 文件时应忽略。
- 如果无关改动影响当前任务，应先说明风险再继续。

## 文档与来源管理

- 赛题 PDF、平台官方资料、第三方仓库和不可读取链接都必须记录来源状态。
- 对未完成实测、未完成归档或未验证的内容，必须明确标注“待测量”“待归档”“待验证”。
- 重量、尺寸、续航、抓取成功率、识别准确率等指标不得凭经验写成已满足。
